# KoreanMate Interview Script

> 목적: KoreanMate Serverless 설계를 면접에서 자연스럽게 설명하기 위한 대본과 예상 Why 질문 답변 정리

---

## 1. 1분 요약 대본

KoreanMate는 한국어 학습자를 위한 AI 기반 학습 서비스입니다. 사용자는 글쓰기 교정, 상황별 회화 생성, 레벨 테스트 기능을 사용할 수 있고, 결과는 학습 기록으로 저장됩니다.

이 프로젝트는 단순히 AI 기능을 구현하는 것보다, Serverless 기반 운영 설계와 비용 제어, 인증/보안, 관측성, CI/CD 자동화를 보여주는 것을 목표로 했습니다.

프론트엔드는 React/Vite로 만들고 S3와 CloudFront로 배포했습니다. S3는 직접 공개하지 않고 CloudFront OAC를 통해서만 접근하도록 구성했습니다.

백엔드는 API Gateway와 Lambda로 구성했습니다. API Gateway는 Cognito JWT Authorizer로 토큰을 검증하고, Lambda는 JWT claims의 `sub` 값을 userId로 사용합니다. 데이터는 DynamoDB에 저장하고, 학습 기록, 사용량 제한, 사용자 프로필 테이블을 분리했습니다.

AI 응답은 Amazon Bedrock을 사용합니다. Bedrock 호출 전에는 사용자별 일일 사용량 제한을 확인해서 AI 호출 비용이 무제한 증가하지 않도록 했습니다.

운영 측면에서는 CloudWatch Alarm, X-Ray, CloudTrail, Grafana Cloud, AWS Budgets를 구성했습니다. 배포는 GitHub Actions에서 CI와 Deploy를 분리하고, Deploy는 수동으로 `plan-only` 또는 `apply`를 선택할 수 있게 했습니다.

## 2. 3분 설명 대본

KoreanMate는 한국어 학습자를 위한 AI 서비스입니다. 주요 기능은 글쓰기 교정, 회화 생성, 레벨 테스트입니다. 저는 이 프로젝트를 클라우드/데브옵스 포트폴리오로 만들었기 때문에, 단순 CRUD보다 운영 설계와 비용 최적화, 보안, 관측성을 중점으로 설계했습니다.

먼저 인프라는 Terraform으로 관리했습니다. 모듈은 API Gateway, Lambda, DynamoDB, Cognito, IAM, S3/CloudFront, CloudTrail, Monitoring, Budget, Grafana Cloud로 나누었습니다. 이렇게 분리한 이유는 리소스 책임을 명확히 하고, 나중에 dev/prod 환경으로 확장할 때 같은 모듈을 재사용하기 위해서입니다.

프론트엔드는 S3와 CloudFront로 배포했습니다. React/Vite는 정적 파일로 빌드되기 때문에 EC2나 컨테이너로 서빙할 필요가 없다고 판단했습니다. S3는 Public Access Block을 적용했고, CloudFront OAC를 통해서만 접근하도록 해서 버킷을 직접 공개하지 않았습니다. CloudFront 앞단에는 WAF를 연결해 기본적인 요청 필터링과 Rate Limit을 적용했습니다.

인증은 Cognito를 사용했습니다. 사용자가 로그인하면 Cognito가 JWT를 발급하고, 프론트엔드는 API 요청에 이 토큰을 포함합니다. API Gateway는 JWT Authorizer로 토큰을 검증한 뒤 Lambda를 호출합니다. Lambda에서는 클라이언트가 전달한 userId를 믿지 않고, Cognito JWT claims의 `sub` 값을 userId로 사용합니다.

백엔드는 기능 단위 Lambda로 분리했습니다. correction, conversation, level-test, profile, history, usage로 나누었고, 각 함수는 Node.js 20 런타임을 사용합니다. Lambda는 CloudWatch Log Group을 가지고 있고, 로그 보관 기간은 14일로 제한했습니다. X-Ray Active Tracing도 활성화했습니다.

데이터는 DynamoDB에 저장했습니다. LearningRecords는 userId와 recordId를 Key로 사용해서 사용자별 학습 기록을 저장합니다. UsageLimits는 userId와 usageDate를 Key로 사용해서 일일 사용량 제한을 관리하고, TTL을 적용해 오래된 사용량 데이터가 자동 삭제되도록 했습니다. UserProfiles는 userId 기준으로 사용자 설정을 저장합니다.

Bedrock 호출 비용을 제어하기 위해 사용량 제한을 직접 구현했습니다. API Gateway Throttling이나 WAF는 요청량 제한에는 좋지만, 사용자별 하루 교정 횟수나 레벨 테스트 횟수 같은 비즈니스 기준 제한은 직접 구현해야 했습니다. 그래서 Bedrock 호출 전에 UsageLimits 테이블을 확인하고, 한도를 초과하면 429를 반환하도록 했습니다.

운영 측면에서는 CloudWatch와 CloudTrail, Grafana를 역할별로 분리했습니다. CloudWatch는 Lambda Error, Duration, API Gateway 5XX를 감지하고, CloudTrail은 AWS API 호출 이력을 감사 로그로 남깁니다. Grafana Cloud는 CloudWatch 지표와 로그를 시각화하기 위해 AssumeRole 방식으로 연동했습니다.

CI/CD는 GitHub Actions로 구성했습니다. CI는 PR에서 빌드, 타입 체크, Terraform validate, Trivy 보안 스캔을 수행합니다. Deploy는 수동 실행만 가능하고, `plan-only`와 `apply`를 선택할 수 있습니다. 실제 AWS 리소스 변경은 apply를 선택한 경우에만 수행되도록 해서 실수로 인프라가 변경되는 것을 막았습니다.

현재 개선할 점도 있습니다. Bedrock IAM 권한은 dev 환경에서는 `Resource = "*"`로 되어 있어 운영 환경에서는 모델 ARN으로 제한해야 합니다. GitHub Actions 인증도 현재는 Access Key 방식이므로 향후 OIDC 기반 AssumeRole로 바꾸는 것이 더 안전합니다. 또한 사용량 증가와 학습 기록 저장은 별도 DynamoDB 작업이라 운영 수준에서는 TransactWriteItems를 검토할 수 있습니다.

## 3. 예상 Why 질문 답변

## Q1. 왜 Serverless를 선택했나요?

개인 포트폴리오 프로젝트는 상시 트래픽이 많지 않고, 서버 운영보다 클라우드 설계와 자동화 역량을 보여주는 것이 더 중요하다고 판단했습니다. Serverless를 사용하면 유휴 비용을 줄일 수 있고, 서버 패치나 용량 관리 부담도 줄어듭니다. KoreanMate는 요청이 들어올 때만 API가 실행되면 되는 구조라 Lambda, API Gateway, DynamoDB 조합이 적합했습니다.

---

## Q2. 왜 S3 + CloudFront로 프론트엔드를 배포했나요?

React/Vite는 빌드하면 정적 파일이 생성됩니다. 정적 파일을 제공하기 위해 EC2나 컨테이너를 운영하는 것은 비용과 운영 부담이 불필요하게 커집니다. S3는 정적 파일 저장에 적합하고, CloudFront는 캐싱과 HTTPS 접근 지점을 제공합니다. 또한 OAC를 사용하면 S3를 직접 공개하지 않고 CloudFront를 통해서만 접근하게 만들 수 있어 보안 측면에서도 적합했습니다.

---

## Q3. 왜 API Gateway + Lambda 구조를 사용했나요?

API Gateway는 인증, CORS, 라우팅을 처리하고 Lambda는 기능별 비즈니스 로직을 실행합니다. KoreanMate의 API는 항상 실행 중인 서버가 필요하지 않기 때문에 요청 기반으로 실행되는 Lambda가 비용 면에서 적합했습니다. 또한 correction, conversation, level-test처럼 기능별 Lambda로 분리하면 장애 범위와 배포 단위를 작게 가져갈 수 있습니다.

---

## Q4. 왜 Cognito JWT Authorizer를 사용했나요?

Lambda에서 직접 JWT를 검증하면 모든 Lambda에 인증 로직이 중복됩니다. API Gateway JWT Authorizer를 사용하면 API 진입점에서 토큰을 먼저 검증할 수 있고, Lambda는 검증된 요청만 처리하면 됩니다. 또한 Lambda에서는 Cognito claims의 `sub` 값을 userId로 사용하기 때문에 사용자가 요청 Body에 임의의 userId를 넣어 다른 사용자 데이터에 접근하는 문제를 줄일 수 있습니다.

---

## Q5. 왜 DynamoDB에 userId 기반으로 저장했나요?

이 서비스의 주요 조회 패턴은 “내 학습 기록”, “내 오늘 사용량”, “내 프로필”입니다. 모두 특정 사용자 기준 조회이기 때문에 userId를 Partition Key로 사용하는 것이 자연스럽습니다. Cognito `sub`를 userId로 사용하면 인증 정보와 데이터 저장 기준이 일치하고, 사용자별 데이터 분리도 명확해집니다.

---

## Q6. 왜 사용량 제한을 직접 구현했나요?

Bedrock은 호출량에 따라 비용이 발생합니다. WAF나 API Gateway Throttling은 IP나 요청량 제한에는 유용하지만, “사용자별 하루 교정 10회”, “레벨 테스트 5회” 같은 서비스 정책은 직접 구현해야 합니다. 그래서 UsageLimits 테이블을 만들고, Bedrock 호출 전에 사용자별 사용량을 확인하도록 했습니다. 이렇게 하면 한도 초과 요청은 Bedrock을 호출하지 않기 때문에 AI 비용을 제어할 수 있습니다.

---

## Q7. 왜 Bedrock Model ID를 SSM Parameter Store로 분리했나요?

Model ID를 코드에 하드코딩하면 모델을 바꿀 때 코드 수정과 재배포가 필요합니다. SSM Parameter Store에 모델 ID를 관리하면 환경별로 다른 모델을 사용할 수 있고, 설정값을 인프라 코드에서 관리할 수 있습니다. 현재 구현은 SSM에 저장한 값을 Terraform이 Lambda 환경변수로 전달하는 구조입니다. 즉 Lambda가 매 요청마다 SSM을 읽는 구조는 아닙니다.

---

## Q8. 왜 CloudWatch, CloudTrail, Grafana를 분리했나요?

세 도구는 목적이 다릅니다. CloudWatch는 애플리케이션 로그와 메트릭, 알람을 담당합니다. CloudTrail은 AWS 계정에서 누가 어떤 API를 호출했는지 감사 이력을 남깁니다. Grafana는 CloudWatch 데이터를 보기 쉽게 시각화하는 대시보드 역할입니다. 그래서 운영 감지는 CloudWatch, 감사 추적은 CloudTrail, 시각화는 Grafana로 역할을 분리했습니다.

---

## Q9. 왜 GitHub Actions에서 CI와 Deploy를 분리했나요?

CI는 코드가 안전한지 검증하는 단계이고, Deploy는 실제 AWS 리소스를 변경하는 단계입니다. 둘을 합치면 PR 검증 과정에서 실수로 인프라가 변경될 수 있습니다. 특히 Terraform Apply는 비용이 발생하는 리소스를 만들 수 있기 때문에 수동 실행으로 분리했습니다. Deploy에서도 `plan-only`와 `apply`를 나누어, 변경사항 확인과 실제 적용을 구분했습니다.

---

## Q10. 비용 최적화는 어디에 반영했나요?

첫째, 상시 서버를 두지 않고 Lambda, API Gateway, DynamoDB, S3, CloudFront 중심의 Serverless 구조를 사용했습니다. 둘째, DynamoDB는 PAY_PER_REQUEST 방식으로 초기 트래픽 예측 없이 시작했습니다. 셋째, Bedrock 호출 전 사용량 제한을 확인해 AI 비용을 제어했습니다. 넷째, CloudWatch Log Retention과 CloudTrail Lifecycle을 설정해 로그 비용을 줄였습니다. 마지막으로 AWS Budgets를 설정해 월간 비용 초과를 감지하도록 했습니다.

---

## 4. 다이어그램 관련 답변

## Q. 이 다이어그램은 문제 없나요?

전체적으로 설계서에 넣어도 됩니다. 다만 API Flow 다이어그램에서 “Read Model Config → SSM Parameter Store”는 현재 구현과 조금 다릅니다. 실제로는 SSM Parameter Store에 Bedrock Model ID를 저장하고, Terraform이 그 값을 Lambda 환경변수로 전달합니다. 그래서 다이어그램 문구를 “Model Config managed by SSM / provided as Lambda env”로 바꾸는 것이 더 정확합니다.

CI/CD 다이어그램은 괜찮습니다. 다만 Deploy Pipeline 안의 “Terraform Deploy”는 면접 설명 기준으로 “Terraform Plan / Apply”라고 표현하면 현재 workflow의 `plan-only`와 `apply` 구조를 더 잘 보여줄 수 있습니다.

전체 아키텍처와 사용자 흐름 다이어그램은 포트폴리오 문서에 넣어도 됩니다.

---

## 5. 마지막 정리 멘트

KoreanMate는 단순히 AI API를 붙인 앱이 아니라, AI 호출 비용을 제어하고, 인증된 사용자 기준으로 데이터를 분리하고, Serverless 운영 환경에서 배포/보안/관측성/감사를 함께 고려한 프로젝트입니다. 현재는 dev 환경 기준으로 구현했고, 운영 수준으로 확장한다면 Bedrock IAM 권한 제한, GitHub OIDC 전환, 알림 채널 연결, DynamoDB 트랜잭션 적용을 개선할 계획입니다.
